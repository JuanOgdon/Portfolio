# 2D Rocket TVC Attitude Control Simulation

**Course:** Control Systems — Final Integrating Project, UTN (2024)
**Co-author:** Felipe Costalonga

This project is a complete 2D simulation of a Thrust Vector Control (TVC) rocket, built in Simulink from physical first principles; built from scratch. The goal was not just to make something that "works," but to understand every layer of the system — from basic kinematics, through full aerodynamic modeling, to a tuned guidance and control loop.

All 8 planned milestones were completed.

---

## Methodology: build complexity incrementally

The entire project was developed milestone by milestone, with each stage validated before the next was added. This wasn't just a workflow choice — it's an engineering discipline. Adding multiple unknowns at once makes it nearly impossible to isolate where a problem comes from.

The 8 milestones were:

1. Open-loop, vertical thrust only — verify the equations of motion produce a parabola
2. Add horizontal force component — verify 2D behavior with a fixed gimbal angle
3. Add real mass and moment of inertia from OpenRocket (m = 1.267 kg, I = 0.016 kg·m²)
4. Add moment arm geometry (CG–CP distance, TVC-to-CG distance = 0.153 m)
5. Add axial drag, modeled and validated against expected terminal velocity behavior
6. Add lift and aerodynamic restoring moment — validated passive stability
7. Close the attitude loop with a PID controller
8. Add waypoint guidance (Waypoint Guidance method) and tune the full system

Each milestone had defined expected behavior. The system was only considered ready to advance when the output matched prediction.

---

## Physical modeling

Real rocket parameters were extracted from two tools:

- **OpenRocket**: designed a 40 cm rocket with two AeroTech G12 motors (8.51 s burn time, ~30 N thrust, 1267 g total). Extracted CG (21.8 cm), CP (24.7 cm), mass, and moment of inertia.
- **RASAero II**: modeled exact geometry to obtain Cl and Cd curves vs. angle of attack across Mach numbers.

One challenge was that the aerodynamic coefficients are continuous functions of alpha, not constants. Rather than using lookup tables, I fitted Cd(α) with a quadratic using Mathematica:

```
Cd(α) = 0.4077 + 0.0018·α²
```

This matched the RASAero II data closely and could be implemented directly in Simulink as a simple algebraic block.

Cl was modeled as linear with alpha (slope ~0.1/degree), consistent with theoretical expectations for a slender body at small angles of attack.

---

## A subtle problem: coordinate frames

Simulink's 3DOF (Body Axes) block uses a convention where Z points **downward**, while the physical intuition (and the force diagrams) use Z pointing up. This required careful sign handling throughout — getting this wrong produces a rocket that accelerates into the ground or an attitude controller that destabilizes instead of stabilizes.

The fix was to explicitly track the sign convention and verify it at each milestone by checking that the model reproduced the expected qualitative behavior (e.g., a parabolic trajectory in Z vs. t for purely vertical thrust, before adding any other complexity).

---

## Constructive misalignment: a physically important result

One intermediate result was particularly instructive. Simulating a 1 mm misalignment of the nozzle axis relative to the rocket's longitudinal axis — with no active control — showed the rocket hitting the ground approximately 2 seconds after launch, despite having 6.5 seconds of remaining burn time. The moment generated by even a small misalignment is large enough to be catastrophic without active control.

This is why TVC exists. Seeing it in the simulation, with real physical parameters, made it concrete.

## PID tuning

Initial PID gains were computed using the Ziegler-Nichols method (no-overshoot variant):

- K_u = 24, T_u = 0.565
- Z-N gives: Kp = 14.4, Ki = 50.97, Kd = 1.017

These were a reasonable starting point, but produced significant steady-state error and a long transient. Manual iterative tuning, guided by knowledge of how each gain affects the response, yielded the final values:

```
Kp = 5,  Ki = 10.95,  Kd = 2.36
```

The lesson: systematic methods give you a first approximation. Getting the best performance requires understanding the system well enough to know which direction to push each parameter and why.

---

## Guidance: Waypoint method

The guidance law computes a desired attitude angle θ_g pointing toward the current waypoint:

```
θ_g = arctan((x_goal - x_rocket) / (z_goal - z_rocket))
```

The PID controller then drives the actual attitude θ to match θ_g. Two abort conditions were implemented:
- z ≤ 0 (ground contact)
- |α| > 30° (angle of attack exceeds linear aerodynamics region — abort for model validity)

Gimbal was hardware-limited to ±5° with a max angular rate of ±5°/s.

---

## Final results

Final simulation with real rocket parameters, 5 mm nozzle misalignment, 8.51 s burn time, and two sequential waypoints:

- Waypoint 1 (200 m horizontal, low altitude): final position error **0.38 m**
- Waypoint 2 (600 m horizontal, 400 m altitude): final position error **2.28 m**

## What I learned

- **Incremental complexity works.** Every bug was caught early, at the simplest version of the system where the expected behavior was unambiguous.
- **Physical data beats assumptions.** Using real OpenRocket/RASAero parameters instead of made-up values meant the simulation reflected the actual vehicle, not a toy problem.
- **Ziegler-Nichols is a starting point, not an answer.** The method gave reasonable initial gains, but the real work was understanding *why* the system behaved the way it did and tuning from that understanding.
- **Coordinate frames are a real engineering problem.** Getting them wrong silently produces wrong results. Checking every transformation against physical intuition is not optional.
- **Small misalignments matter enormously.** A 1 mm nozzle offset in a real rocket is the difference between a controlled flight and a crash — the simulation made this viscerally clear.
